0. Executive Summary

The app is a multi-tenant B2B distributor ERP for a New Zealand health-products company. It is approximately 45 pages, 118 server actions across 15 action files, 15 distinct reports (CSV/XLSX/PDF), 30+ Prisma models, 4 daily cron jobs pinned to NZT, and a QuickBooks Online integration with AES-256-GCM-encrypted refresh tokens. Traffic is small (low tens of users), but report generation is CPU-heavy and transactional writes (inventory moves, reservations, landed-cost allocation) must be correct.

The current deployment works but has three material problems:

Three production gaps in the current stack

  1. ALB is HTTP only, not HTTPS. Cookies, passwords, and QBO OAuth tokens traverse the public internet unencrypted.
  2. Cron jobs are declared in vercel.json but the app runs on AWS ECS. Daily FX refresh, low-stock alerts, QBO sync, and scheduled reports never fire.
  3. Secrets live in plain ECS task-definition env vars. SESSION_SECRET, QBO_ENCRYPTION_KEY, DATABASE_URL are readable by anyone with ecs:DescribeTaskDefinition.

Before recommending a rebuild, I note these three issues can all be fixed in a one-week incremental pass on the current stack. A rebuild is only justified if we also want lower TCO, better cold-start economics, cleaner separation of concerns, or a different cloud for procurement reasons.

Final recommendation

A AWS Refined Gold pick

ECS Fargate + HTTPS ALB + RDS + EventBridge Scheduler + Secrets Manager + S3. Lowest migration cost, retains existing SST/Pulumi IaC, closes all three production gaps, preserves Next.js server actions.

~$95/mo ~5 days to ship Zero code rewrite

B GCP Cloud Run Silver pick

Cloud Run + Cloud SQL + Cloud Scheduler + Secret Manager + Cloud Storage. Scale-to-zero economics, simplest operator model, managed HTTPS out of the box.

~$55/mo ~5–7 days to ship Cold start <1 s

2. Current Architecture — Honest Assessment

Current topology

Current deployment in ap-southeast-2. Dashed red/amber boxes flag the production gaps.

What works

  • Single codebase, single deployment unit. Next.js + Prisma on one container.
  • Server Actions + Prisma combo is highly productive for CRUD-heavy ERP — 118 actions across 15 files.
  • Multi-tenant enforcement consistent — every action starts with requireSession() and scopes by tenantId.
  • Schema is clean — 30 models, proper tenantId indexes.
  • IaC is TypeScript (SST/Pulumi). arm64 Fargate is ~20% cheaper than x86.

App profile — what matters for architecture choice

  • Low concurrent load — single-digit users typically.
  • Bursty CPU — report gen 1–5 s of @react-pdf / ExcelJS work.
  • Transactional writes — stock moves, reservations, landed cost must be ACID.
  • Scheduled compute — four fixed NZT crons daily.
  • Data-critical ledger — the business lives in Postgres.
  • Single locality — all users NZ/AU. No global CDN value.
  • Small team — 1 dev. Ops overhead must be ≈ zero.

3. Five Candidate Architectures

Each option: topology, services, what changes, strengths, weaknesses, indicative monthly cost. Costs are on-demand list prices for ap-southeast-2 / australia-east / australia-southeast1 with no commitments.

1 AWS Refined — Container + RDS, done properly Recommended · Gold

Same shape as today, every rough edge removed.

Option 1 topology — existing shape with HTTPS, private DB, Secrets Manager, real cron, CloudWatch.

Strengths

  • Smallest migration distance. Server Actions, Prisma, domain code unchanged.
  • Retains team know-how; developer keeps AWS / SST / Pulumi.
  • Closes every known production gap in one release.
  • Multi-AZ RDS removes single-zone outage risk.

  • Always-on Fargate — pays for idle compute 24/7.
  • Four-hop edge (Route 53 → CloudFront → ALB → Fargate) adds ~30 ms.

  • Broad operational surface area (10+ services in IaC).

2 AWS Serverless — OpenNext + Aurora Serverless v2

Next.js runs on Lambda via OpenNext; Postgres is Aurora Serverless v2; no always-on compute.

Strengths

  • True per-request compute billing; idle nights ≈ $0.
  • Automatic scale-out for bursts.
  • No container image to maintain.
  • Aurora Serverless v2 scales to hundreds of tenants without re-architecting.

Weaknesses

  • Cold starts: 1–3 s for first hit after idle.
  • Aurora floor: 0.5 ACU min ≈ $43/mo (cannot pause).
  • Prisma-on-Lambda quirks (connection limit, binary target).
  • OpenNext is an extra abstraction that can break on Next.js minor versions.
  • Observability fragmentation across Lambda log groups.

Indicative total: ~$70 / month (mostly Aurora + Proxy).

3 Azure Container Apps — Microsoft Stack

Container Apps (scale-to-zero) + Flexible Postgres + Container Apps Jobs for crons.

Strengths

  • Scale-to-zero — idle overnight = storage only.
  • Managed HTTPS via Front Door is a one-liner.
  • CA Jobs is a cleaner cron primitive than EventBridge → RunTask.
  • Enterprise fit: if NZ targets are M365-aligned, Entra ID SSO is easy.
  • B1ms DB is cheap (~$18/mo).

Weaknesses

  • Cold starts 1–3 s on scale-from-zero.
  • Team has no stated Azure experience — ~1 week ramp.
  • Front Door egress is higher than CloudFront at low volume.
  • Private endpoint + VNet setup is more wizardry than AWS.
  • Fewer reliable AI-agent MCPs, more portal clicks.

Indicative total: ~$40–74 / month depending on Front Door inclusion.

4 GCP Cloud Run + Cloud SQL — Simplest, Lowest TCO Recommended · Silver

Cloud Run containers + Cloud SQL + Cloud Scheduler pulling HTTPS endpoints.

Strengths

  • Scale-to-zero with industry-leading cold starts (200–800 ms).
  • Managed HTTPS is a one-liner; certs auto-issue.
  • Cloud Scheduler is the cleanest cron primitive of the five.
  • Lowest idle cost: ~$30/mo at genuinely low usage.
  • Simplest operator model for a 1-dev team.

Weaknesses

  • Cloud SQL connector / VPC connector adds a step.
  • Less NZ SME procurement familiarity than AWS/Azure.
  • No NZ region — Sydney (australia-southeast1) is nearest.
  • Team has no stated GCP experience — ~1 week ramp.

5 Split Architecture — Frontend + API + Queue Worker

Static frontend + dedicated API + background worker + queue. Cloud-agnostic.

Strengths

  • Independent scaling — a slow report in the worker never blocks users.
  • Cloud-portable — same images run on AWS, Azure, GCP.
  • Clean REST surface is easier to test than scattered Server Actions.
  • Long-running report jobs in worker, not HTTP handler.

Weaknesses

  • Major rewrite — 8–12 weeks. Every Server Action becomes a REST endpoint.
  • Higher infra cost (~$145+/mo) — 3 containers + Redis + queue.
  • Two codebases minimum; auth wired twice.
  • Benefits only materialise at scale this app doesn't have.
  • Abandons the productive Server Actions + Prisma model.

Indicative total: ~$145+ / month on AWS.

5. Final Recommendation — Top Two

Given the app shape (CRUD-heavy, low traffic, bursty CPU on reports, 1-dev team, data-critical ledger, NZ locality), the two strongest options are:

A AWS Refined Gold — fastest path to production-safe

Best for: the fastest path to a safe, correct production system with minimal risk.

  • Keeps 100% of the existing domain code intact.
  • Fixes every production gap (HTTPS, secrets, real crons, private DB, migrations, CI/CD, backups, alarms) in a single focused ~5-day effort.
  • The existing SST/Pulumi IaC stays; only deltas are added.
  • Ongoing cost is ~$95/mo single-AZ or ~$131/mo multi-AZ, predictable and flat.
  • No cold starts. Report pages stay snappy.
  • Team retention: developer continues using the AWS tools they already know.

Pick over B if: you want to ship a hardened production system in a week, you don't want to learn a new cloud, you're prepared to pay for always-on compute.

B GCP Cloud Run Silver — lowest TCO

Best for: long-term lowest TCO with the simplest operator model.

  • Scale-to-zero: idle overnight costs pennies.
  • Fastest cold starts of any scale-to-zero option (200–800 ms typical).
  • Managed HTTPS and custom-domain certs are one-liners.
  • Cloud Scheduler is the cleanest cron primitive of the five options.
  • The operator surface is smaller than AWS — fewer services, simpler IAM.
  • Ongoing cost at low usage: ~$30–65/mo depending on whether the Global HTTPS LB is used.

Pick over A if: you want the lowest possible monthly bill, you're comfortable learning GCP, you value operational simplicity over platform familiarity, procurement has no AWS mandate.

Why not the others

  • Option 2 (AWS Serverless): Tempting for scale-to-zero, but Aurora Serverless v2's $43/mo floor eliminates the cost advantage at this scale, and OpenNext adds a debugging layer the team doesn't need. If traffic grows 10× this becomes the best option.
  • Option 3 (Azure Container Apps): Technically excellent, competitive on price. Recommend over GCP only if there's a Microsoft 365 / Entra ID procurement reason.
  • Option 5 (Split): The right answer if this app were 10× bigger, multi-team, or multi-product. Wrong answer for a 1-dev team shipping an inventory ERP. Revisit in 18 months if the business warrants it.

6. Rebuild Prompt — Option A (AWS Refined)

Hand this brief to a development team — fresh or continuing — to rebuild the app on AWS with every gap closed.

Rebuild Brief: NZ Inventory ERP on AWS (Fargate + RDS + HTTPS + EventBridge)

Product: Multi-tenant B2B distributor ERP for a New Zealand health-products company. Users manage products, stock, batches, suppliers, customers, purchase orders, sales orders, proforma invoices, credit notes, reservations, 15 reports, and a QuickBooks Online sync. Current users are in the low tens; architecture must support 10×.

Goal: Production system on AWS ap-southeast-2, HTTPS end-to-end, private database, managed secrets, real scheduled jobs, safe migrations, CI/CD, backups, and alarms. Preserve the existing codebase; change only what must change.

Non-goals: Rewriting Server Actions, changing Prisma, introducing microservices, adding a queue, replacing the frontend.

Source of truth: Existing Next.js 15 App Router + Prisma 5 + PostgreSQL repo. Retain every page, every server action, every Prisma model, every report, every API route, all iron-session auth logic, all fiscal-year / NZT behaviour, and the QuickBooks Online integration with AES-256-GCM-encrypted refresh tokens.

Target Architecture

  • Region: ap-southeast-2.
  • Edge: Route 53 hosted zone → CloudFront distribution → ALB on HTTPS only (:443). ALB :80 redirects to :443. ACM cert auto-renewed. HSTS header set by the app.
  • Compute: ECS Fargate, arm64, 0.5 vCPU / 1 GB, desired 1, min 1, max 3. Health check /api/health. Task execution role reads from Secrets Manager. Task role has least-privilege S3 + Secrets Manager access only.
  • Database: RDS PostgreSQL 16, db.t4g.small multi-AZ, 50 GB gp3, automated backups 7 days, PITR on, private subnets, publicly-accessible=false, deletion protection on, Performance Insights on.
  • Networking: VPC with 2 public subnets (ALB, NAT) and 2 private subnets (Fargate, RDS) across 2 AZs. One NAT Gateway. SGs: ALB 80/443 public; Fargate 3000 from ALB SG only; RDS 5432 from Fargate SG only.
  • Secrets: Secrets Manager — one secret per value: session-secret, qbo-client-id, qbo-client-secret, qbo-encryption-key, database-url, resend-api-key, cron-secret. Wired via task-def secrets block.
  • Storage: S3 bucket nz-inv-artifacts-<account>-<region>. SSE-S3, public access blocked, versioned. Folders reports/{tenantId}/{date}/, product-images/{tenantId}/, invoices/{tenantId}/. CloudFront OAI for signed-URL delivery.
  • Cron: 4× EventBridge Scheduler — each cron(... Australia/Auckland):
    • qbo-sync 08:00 NZT → ECS RunTask with nz-inv-cron task def, env CRON_HANDLER=qbo-sync.
    • low-stock 19:00 NZT.
    • fx-rates 18:00 NZT.
    • reports 07:00 NZT.
    Cron task def mirrors web task def but overrides entrypoint to node scripts/cron-handler.js.
  • Registry: ECR immutable tags git-<sha>. Lifecycle keeps last 20.
  • CI/CD: GitHub Actions on main — build arm64 image, ECR push with commit SHA, aws ecs update-service --force-new-deployment, wait for steady state, smoke-test /api/health.
  • Observability: CloudWatch Logs retention 30 d on Fargate + cron log groups. Alarms on ALB 5xx >1%, Fargate CPU >80%, Memory >80%, RDS storage <10% free, RDS CPU >80%, cron failures. Minimal dashboard. Container Insights optional.
  • Migrations: Drop prisma db push --accept-data-loss. Commit migrations to prisma/migrations/. CI runs prisma migrate deploy as one-shot ECS task before rolling the web service.
  • IaC: Keep SST 4.7 + @pulumi/aws in TypeScript. One sst.config.ts expressing every resource above. sst deploy --stage prod idempotent.

Code changes required (minimum set)

  1. lib/session.ts — remove dev fallback path in prod; SECURE_COOKIE keyed on APP_URL starts with https://.
  2. lib/storage.ts (new) — S3 client; putReport, getReportSignedUrl, putProductImage. Replace all lib/supabase/admin.ts usage.
  3. lib/secrets.ts (new, optional) — at boot, read secrets from env (injected by ECS).
  4. scripts/cron-handler.ts (new) — switch (process.env.CRON_HANDLER) { ... } dispatching to the four existing route handlers.
  5. app/api/cron/* — keep for manual invocation; make handlers importable so cron-handler can call them directly.
  6. vercel.json — delete.
  7. middleware.ts — no change.
  8. Dockerfile — drop prisma db push from startup CMD; startup is node server.js.
  9. package.json — add "migrate:deploy": "prisma migrate deploy".
  10. .github/workflows/deploy.yml (new) — checkout, docker buildx arm64, ECR push, prisma migrate deploy one-shot ECS task, aws ecs update-service, smoke test.

Acceptance criteria

  • curl -I https://<domain>/ returns HTTP/2 307 redirect to /login.
  • curl -I http://<domain>/ returns 301 to https://<domain>/.
  • SSL Labs scores A or better.
  • Direct Postgres from public internet times out.
  • Task definition JSON shows no plain secret values; Secrets Manager returns them.
  • All 4 EventBridge schedules visible; manual fire writes ExchangeRate / sends alert / drains QboSyncJob / writes ReportDelivery.
  • prisma migrate status clean; no --accept-data-loss anywhere.
  • Bad deploy rollback via aws ecs update-service --task-definition <previous> in <2 min.
  • CloudWatch dashboard live; synthetic alarm fires to SNS.

Deliverables

  • sst.config.ts with full resource graph.
  • Cleaned Dockerfile.
  • lib/storage.ts, scripts/cron-handler.ts, lib/secrets.ts.
  • GitHub Actions workflow.
  • Data migration from current RDS to new RDS (pg_dump / pg_restore, documented runbook).
  • Runbooks: docs/runbooks/deploy.md, rotate-secrets.md, backup-restore.md, incident-response.md.
  • One-page architecture diagram & cost sheet.

Constraints

  • Preserve all 15 reports and their exports — no regression in PDF/XLSX output.
  • Preserve iron-session cookie auth. Do not introduce Cognito.
  • Preserve multi-tenant isolation through app code; no RLS in this pass.
  • Preserve NZT fiscal-year logic (April–March) and all date formatting.
  • Preserve QBO OAuth flow and AES-256-GCM refresh-token encryption.
  • Don't touch Prisma models unless adding a migration-safe constraint.
  • No Vercel.

Estimated effort

Experienced AWS engineer familiar with Next.js + SST: 5–7 working days.

8. Appendix — Honest Caveats

  • No hallucinated features. Every capability listed in Section 1 maps to real files in the repo.
