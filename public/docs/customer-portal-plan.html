Executive summary

If you only read one box, read this one.

★ Recommended

Embedded portal in the existing app, invite-only signup, TOTP 2FA from day one

Add a new route group app/(portal)/… living next to the internal app, sharing the database but with strict customer-scoped guards. Customers log in with email + password, set up an authenticator-app code on first login, and place draft orders that your sales team confirms before anything ships.

  • Scope: ~6 days for Phase 1 (auth + catalogue + cart + draft orders + history). 2FA is part of Phase 1, not bolted on later.
  • Infra: reuses your AWS ECS deployment, your Postgres, your iron-session — no new services, no monthly fees.
  • Safety posture: TOTP 2FA, 8 single-use backup codes, password rules, rate-limited login, audit log of every auth event, invite-only signup.
  • Order policy: portal customers can only create DRAFT SOs — internal staff confirm and ship. No "self-checkout-and-it's-gone" risk.

The plan below explains why this option beats the alternatives, walks you through every screen the customer (and your admin) will see, and lays out the safety story in plain English. Five other approaches are compared — headless storefronts, magic links, customer SSO, separate subdomains, read-only-first — with the trade-offs that pushed each below the chosen one.

Today's reality

How orders flow now, and what changes.

How an order happens today

A buyer at Chemist Warehouse picks up the phone, emails the rep, or pings them in WhatsApp. Your sales rep at Regional Health logs into the internal app, opens Sales Orders → New SO, picks the customer, types the SKUs, and the form auto-applies that customer's price-group pricing. Rep saves the SO; warehouse picks; rep ships; QBO sync runs.

This works, but it's friction for both sides:

  • The customer can't see what's in stock without asking.
  • They can't see their own order history.
  • Every order needs a rep to be available to take it.
  • Slips happen — rep mistypes a SKU, prices drift, units get confused.

What the portal changes

Customer logs in, sees their catalogue at their price, picks quantities, submits. A new DRAFT sales order lands in your internal app — your sales team gets a notification, reviews it, confirms, and ships. Same final state as today; just half the friction and a much better customer experience.

Critical design choice: portal customers create draft orders only. Confirmation and fulfilment stay with your team. You keep control of pricing mistakes, credit limit checks, and stock reservation rules — and you can still run a phone-and-email order alongside if you want. The portal is additive, not a replacement.

Architecture at a glance

One Next.js app, two route groups, one database — but with strict isolation between customer and staff.

  • Two cookies, two sessions. Staff use nz_inv_session (existing). Portal users use nz_inv_portal. They can't impersonate each other; logging into one doesn't log you out of the other.
  • One database, scoped queries. Every portal query goes through requirePortalSession() which returns { customerId }. Every read/write is filtered by that customerId. assertCustomer(resourceCustomerId, sessionCustomerId) mirrors the existing tenant guard pattern.
  • Same deploy pipeline. Same Docker image, same ECS service, same RDS. No new infra to monitor.

Safety & 2FA, in plain English

A defensible posture for a B2B portal that touches real money.

The five safety layers

LayerWhat it doesStandard
Password Stops casual attackers and credential reuse from prior breaches. bcrypt cost 12, min 12 chars, HIBP-checked
TOTP 2FA Stops anyone holding a stolen password — they also need the customer's phone. RFC 6238 (Google Authenticator standard), SHA-1, 6 digits, 30-second window
Backup codes Recovery path when the customer loses their phone — and a softer alternative to "email the admin". 8 single-use codes, bcrypt-hashed at rest
Rate limiting Stops brute-force on password and 2FA endpoints. 5 attempts / 15 min / IP (login), 10 attempts / 15 min / account (2FA)
Audit log Both you and the customer can see every login, 2FA attempt, password change, invitation acceptance. Append-only, retention 12 months

Session policy

  • Cookie name: nz_inv_portal (separate from staff cookie nz_inv_session).
  • Flags: HttpOnly · Secure · SameSite=Lax · Path=/.
  • Lifetime: 24h sliding (touched on every authenticated request), 30-day absolute max.
  • Logout: destroys the session server-side and unsets the cookie. Logging out one device doesn't affect others.

Operational guardrails

  • Invite expiry: 24-hour, single-use, replaceable. Expired invites can be reissued from the admin panel.
  • Account lockout: after 10 consecutive failed 2FA attempts in 15 minutes, the account locks for an hour. Admin can clear early.
  • Lost device: use a backup code to sign in, then re-enrol 2FA. If all backup codes are exhausted, an admin can reset 2FA from the internal app — every reset is audit-logged with who-did-it.
  • No self-signup. The login page does not link to a "Create account" page. The only way in is via an admin's invitation.
  • Cost prices stay invisible. Portal queries select sellPriceNzd only. costNzd never leaves the server.
  • Customer scoping. Every portal action begins with const session = await requirePortalSession(); …assertCustomer(row.customerId, session.customerId). Same proven pattern as the staff app's assertTenant.

What 2FA is not: SMS-based. We don't use SMS for 2FA — SIM swap attacks have made it the weakest mainstream factor. Authenticator apps (TOTP) are free, easy, and dramatically stronger.

Future upgrade path: passkeys / WebAuthn

Phase 3 adds support for passkeys (WebAuthn) as a 2FA factor — the user holds their phone or fingerprint reader, and the browser handles the cryptography. No 6-digit code to type, phishing-resistant by design. Once shipped, customers can pick "Authenticator app or passkey" per device. We won't lead with this for v1 because authenticator apps already cover the safety bar and are universally familiar.

Open questions for you

Decide these before any code lands. None are technical blockers — they shape the policy.

  1. One portal user per customer, or many?
    Recommendation: many. Schema supports both at the same cost; supports staff turnover and multi-buyer teams. low risk
  2. Can portal customers edit a draft order after submitting, or only cancel?
    Recommendation: edit while DRAFT, cancel any time before CONFIRMED. After CONFIRMED, only your sales team can change it. low risk
  3. Self-confirm at any value, or always sales-team review?
    Recommendation: always sales-team review for v1. Add an "auto-confirm under $X" toggle later if Chemist Warehouse asks. policy choice
  4. Should portal customers see your stock counts (with 0 = "out of stock"), or just "available / not available"?
    Recommendation: show actual counts — they're useful for procurement planning, and you're not exposing anything sensitive. low risk
  5. QBO sync at submit, or at confirm?
    Recommendation: at confirm. Drafts shouldn't generate QBO invoices. low risk
  6. Email service: you have resend already in package.json. OK to use for invitations, password resets, 2FA alerts? Otherwise pick another (SES, Postmark). low risk
  7. Subdomain or path? portal.regionalhealth.co.nz vs regionalhealth.co.nz/portal?
    Recommendation: path for v1 (simpler DNS, same SSL cert). Move to subdomain later if you brand the portal separately. low risk
  8. Public-self-signup?
    Recommendation: hard no. Invite-only protects you from spam accounts, scrapers, and competitor reconnaissance. strongly recommended

Appendix · schema sketch

Three new tables. Existing tables untouched.

model CustomerUser {
  id            String   @id @default(cuid())
  customerId    String
  customer      Customer @relation(fields: [customerId], references: [id], onDelete: Cascade)
  email         String   @unique
  passwordHash  String
  name          String?
  // Permissions inside that customer's scope
  canPlaceOrders Boolean @default(true)
  canViewInvoices Boolean @default(true)
  canManageUsers  Boolean @default(false)
  // 2FA
  totpSecretEnc String?              // encrypted at rest
  totpEnabled   Boolean  @default(false)
  backupCodes   BackupCode[]
  // Lifecycle
  invitedById   String?              // User.id of the staff member who invited
  acceptedAt    DateTime?
  lockedUntil   DateTime?
  createdAt     DateTime @default(now())
  updatedAt     DateTime @updatedAt

  sessions      PortalSession[]
  auditEvents   PortalAuditEvent[]
  @@index([customerId])
}

model PortalInvitation {
  id          String   @id @default(cuid())
  customerId  String
  customer    Customer @relation(fields: [customerId], references: [id], onDelete: Cascade)
  email       String
  tokenHash   String                  // single-use, 24h TTL
  invitedById String?
  expiresAt   DateTime
  acceptedAt  DateTime?
  createdAt   DateTime @default(now())
  @@index([customerId, expiresAt])
}

model BackupCode {
  id              String       @id @default(cuid())
  customerUserId  String
  user            CustomerUser @relation(fields: [customerUserId], references: [id], onDelete: Cascade)
  codeHash        String                  // bcrypt
  usedAt          DateTime?
  createdAt       DateTime @default(now())
  @@index([customerUserId])
}

model PortalAuditEvent {
  id              String       @id @default(cuid())
  customerUserId  String?
  user            CustomerUser? @relation(fields: [customerUserId], references: [id], onDelete: SetNull)
  customerId      String?                 // for cross-user "all events for this customer" queries
  type            String                  // SIGN_IN, SIGN_IN_FAIL, 2FA_PASS, 2FA_FAIL, PASSWORD_RESET, etc.
  ip              String?
  userAgent       String?
  meta            Json?
  createdAt       DateTime @default(now())
  @@index([customerUserId, createdAt])
  @@index([customerId, createdAt])
}

No changes to Customer, SalesOrder, Product, etc. The portal is purely additive at the schema level.
