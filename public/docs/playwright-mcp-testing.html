Why this exists

Three problems this system solves at once:

  1. Manual testing forgets things. Playwright MCP runs through any flow you describe, every time, the same way.
  2. Bug reports leak into Slack, sticky notes, and screenshots in folders. Auto-logging puts them in the same in-app tracker the team already opens to triage.
  3. Test scripts rot. There aren't any to maintain — Claude reads your prompt and drives the browser. Re-prompt to refine; archive prompts that proved their worth.

Already in production. The bug-report API, the bearer token, and the Settings → Support tab are deployed to the prod ALB. You can use this guide right now.

How it fits together

Claude Code drives a Chromium browser through Playwright MCP. When it notices something broken, it either POSTs to the bug-report API (token-gated) or fills in the Settings → Support form on screen.

Two paths to log a bug

Use either or both.

Path A · API

Fast & machine-friendly.

  • POST JSON to /api/bug-reports
  • Authorization: Bearer <token>
  • No browser session needed
  • Best for assertions & batch runs
Path B · UI

Dogfoods the same form humans use.

  • Drive Playwright to /settings/support
  • Open the dialog, fill, submit
  • Verifies the UI itself works
  • Slower, but a great smoke test

Get going in four steps

1

Install Playwright MCP into Claude Code

One command — registers the server in ~/.claude.json.

claude mcp add playwright -- npx @playwright/mcp@latest

Restart any active Claude session. Verify with /mcp — playwright should appear, listing tools like browser_navigate, browser_click, browser_type, browser_snapshot.

2

Make the bug-report token reachable

The token is already in .env locally and in the prod ECS task definition. Export it into your shell so commands don't have to hard-code it:

export BUG_REPORT_API_TOKEN="$(grep BUG_REPORT_API_TOKEN '/Users/barnie/Claude AI /Regi/.env' | cut -d'\"' -f2)"

Add the same line to ~/.zshrc so new shells pick it up automatically.

Rotate before sharing this token externally. Regenerate with openssl rand -hex 32, update .env and sst.config.ts, then redeploy the ECS task definition.

3

Verify the connection

Token sanity-check (no DB write):

PROD="http://WebLoadBalancer-tbaambtx-482012902.ap-southeast-2.elb.amazonaws.com"
curl -s -H "Authorization: Bearer $BUG_REPORT_API_TOKEN" "$PROD/api/bug-reports"
# → {"ok":true,"message":"Token valid"}

Log a real bug end-to-end:

curl -s -X POST "$PROD/api/bug-reports" \
  -H "Authorization: Bearer $BUG_REPORT_API_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "description": "Smoke test from Playwright MCP setup",
    "affectedAreas": ["other"],
    "reporter": "playwright"
  }'
# → {"ok":true,"id":"...","createdAt":"...","url":"/settings/support"}

Open Settings → Support in the app — the bug appears in the Open list.

4

Run a real test from a Claude prompt

Once Playwright MCP is connected, prompt Claude with what you want tested. Three example prompts that work today:

A · Login & role gating

Test login on the prod URL. Sign in as owner@regionalhealth.co.nz.
Confirm the sidebar shows all 11 nav items.
If any are missing, log a bug via the API
under "auth" describing which item is missing.

B · PO form auto-fill

On prod, go to Purchase Orders → New, pick a supplier whose
currency is set to USD in the supplier record. Verify Currency
auto-snaps to USD. If it stays NZD, log a bug under
"purchase-orders" with the supplier name and what you saw.

C · Multi-image upload (dogfood)

On a product detail page, switch to Images tab. Upload three
small JPEG images. Verify all three appear, the first has a
green ★ Primary badge, and clicking ★ Primary on a non-primary
image promotes it. If anything fails, drive the UI to
Settings → Support and log a bug with reporter "playwright".

API reference

POST /api/bug-reports · auth: Authorization: Bearer <token>

Body schema

FieldTypeReqNotes
description string ✓ 1–5000 chars, trimmed.
affectedAreas string[] Subset of canonical keys (table below).
driveLink string | null Valid URL when present, max 2048 chars.
reporter string | null Free text, max 120 chars.

Canonical area keys

dashboard · products · inventory · purchase-orders · sales-orders · proforma · reservations · suppliers · customers · reports · settings · mobile · qbo-sync · auth · other

Responses

StatusBodyMeaning
201{ok:true, id, createdAt, url}Bug logged.
400{ok:false, error, fieldErrors}Payload invalid (Zod).
401{ok:false, error:"Unauthorized"}Missing or wrong token.
500{ok:false, error:"Server not configured"}Token env var unset.

Best practices

  • One bug per POST. Don't batch unrelated symptoms — the area-tickbox filter works better with single-symptom rows.
  • Structure descriptions. Multi-line works. Try Steps: / Expected: / Actual: at the top of every report.
  • Drop screenshots in Drive, paste the URL. The driveLink field exists for exactly this; the in-app row shows a one-click "Drive ↗" button.
  • Use reporter consistently. "playwright" for automated runs, your name for manual ones — makes filtering by source trivial.
  • Don't log infra noise. ALB 502s and DB timeouts aren't Regi defects. The tracker is for app-level behaviour.
  • Mark solved fast. The Open filter only stays useful if it's pruned. Toggle the green tick the moment a fix lands.

Never paste the production token into shared docs, screenshots, or external chats. If it leaks, rotate it immediately — both .env and sst.config.ts need to update, then redeploy.
